# Архитектурное соответствие нефункциональным требованиям

| № | Требование | Шаблон / свойство |
|---|---|---|
| 1 | Время авторизации не должно превышать 3 секунд | Stateless backend API + PostgreSQL sessions. Авторизация выполняется одним HTTP-запросом и чтением пользователя/сессии из БД без внешних зависимостей. |
| 2 | Интерфейс списка комнат должен загружаться менее чем за 2 секунды | Reverse Proxy + SPA + indexed read model. React-приложение отдается через nginx, список комнат читается через `GET /rooms`, для публичных активных комнат есть индекс по `visibility`, `active`, `created_at`. |
| 3 | Система должна поддерживать не менее 20 одновременно существующих комнат | Persistent room state. Комнаты хранятся в PostgreSQL как независимые записи, backend не держит их только в памяти. |
| 4 | Система должна поддерживать не менее 15 участников в одной комнате | Transactional consistency + database constraints. Участники хранятся в `room_participants`, дубликаты запрещены составным ключом `(room_id, user_id)`, лимит проверяется в транзакции с блокировкой комнаты. |
| 5 | Отключение одного пользователя не должно разрывать соединение остальных участников | Isolated participant lifecycle. Leave/kick удаляет только конкретного участника, остальные остаются в комнате; если выходит хост, роль передается следующему участнику. |
| 6 | Задержка передачи аудио менее 400 мс | Dedicated media plane. Аудио не проходит через backend API, а передается через LiveKit SFU/WebRTC. |
| 7 | Задержка передачи видео менее 600 мс | SFU architecture. Видео маршрутизируется LiveKit SFU, backend отвечает только за control-plane и выдачу токенов. |
| 8 | Создатель комнаты должен иметь базовые инструменты управления участниками | Role-based access control + server-side enforcement. Backend проверяет роль хоста и применяет mute/kick/end через LiveKit RoomService API. |
| 9 | Вышедший или исключенный пользователь не должен видеть сообщения и события комнаты | Membership-based authorization. Доступ к сообщениям, SSE и media-token проверяет актуальное участие пользователя в комнате. |
| 10 | Чат должен работать во время звонка независимо от качества media-связи | Control-plane / media-plane separation. Чат идет через backend API и SSE, аудио/видео идут через LiveKit. |
| 11 | Система должна предпочитать Availability + Eventual Consistency для realtime-сценариев | Event-driven UI + state resync. UI получает SSE-события, а при потере события может восстановить состояние комнаты через HTTP-запрос. |
| 12 | Система должна быть готова к эксплуатации в контейнерной среде | Health checks + reverse proxy. Есть `/health/live`, `/health/ready`, Docker `HEALTHCHECK`, nginx как единая входная точка. |
